87 /// \brief Parsers for feature models in SXFM.
88 ///
89 /// \section SXFM
90 ///
91 /// SXFM stands for Simple XML Feature Model and represents a format that is
92 /// used to store feature models. In general, feature models consist of a
93 /// feature model in a tree-like format and cross-tree constraints (i.e.,
94 /// constraints that would violate the tree-like feature). The SXFM format also
95 /// distinguishes between a feature tree and cross-tree-constraints. The feature
96 /// tree is embeded as a tabulator indented text in an XML tag called
97 /// \c `feature_tree` whereas the cross-tree-constraints are embeded in XML xml
98 /// tag called \c `constraint`. A brief example and description of the SXFM
99 /// format can be found at SPLOT (go to `Edit an existing model` -> `SXFM
100 /// format`). However, they have not given a formal description of the syntax in
101 /// form of a syntax diagram. Note that the SXFM format does only support binary
102 /// features.
103 ///
104 /// \subsection FeatureTree
105 ///
106 /// In the following, we show the syntax diagram for the syntax of feature
107 /// trees.
108 ///
109 /// \image html feature_tree.png
110 ///
111 /// The item \c `tabulator` stands for the tabulator symbol.
112 ///
113 /// By using the sequence diagram generator, the syntax diagrams can be
114 /// generated when using the following rules:
115 ///
116 /// \code{unparsed}
117 /// macro_rules! sxfm_feature_tree {
118 /// (:r $name:expr $id:ident) => { ... };
119 /// (:r $name:expr) => { ... };
120 /// ($tabulator:item $($tabulator:item)* :o $name:expr $id:ident ) => { ...
121 /// };
122 /// ($tabulator:item $($tabulator:item)* :o $name:expr ) => { ... };
123 /// ($tabulator:item $($tabulator:item)* :m $name:expr $id:ident) => { ... };
124 /// ($tabulator:item $($tabulator:item)* :m $name:expr ) => { ... };
125 /// ($tabulator:item $($tabulator:item)* :g [1..*] ) => { ... };
126 /// ($tabulator:item $($tabulator:item)* :g [1..1] ) => { ... };
127 /// ($tabulator:item $($tabulator:item)* : $name:expr ) => { ... };
128 /// ($tabulator:item $($tabulator:item)* : $name:expr $id:ident ) => { ... };
129 ///}
130 /// \endcode
131 ///
132 /// Note that the tabulators indicate the level of the feature. Only the root
133 /// feature is on level 0 (all other features have a higher indentation). There
134 /// are different kind of features:
135 ///
136 /// \li \c `:r` the root feature
137 /// \li \c `:o` the optional feature
138 /// \li \c `:m` the mandatory feature
139 /// \li \c `:g` the parent node of an or group
140 /// \li \c `:` child of an or group
141 ///
142 /// Note that the or-group features (\c ':g' ) have no name.
143 ///
144 /// \subsection Constraints
145 ///
146 /// \image html constraints.png
147 ///
148 /// or alternatively:
149 /// \code{unparsed}
150 /// macro_rules! sxfm_constraints {
151 /// ($name:ident : $cnf_formula:tt) => {...};
152 /// }
153 /// \endcode
154 ///
155 /// The constraints have to be provided in CNF format.
156 /// Therebey, \c `~` stands for negation, \c `or` for a disjunction.
157 /// Each constraint is written in a separate XML tag \c `constraints`.
158 ///
159 /// @see[SPLOT](http://www.splot-research.org/sxfm.html)
160 /// @see[SequenceDiagramGenerator](https://lukaslueg.github.io/macro_railroad_wasm_demo/)
